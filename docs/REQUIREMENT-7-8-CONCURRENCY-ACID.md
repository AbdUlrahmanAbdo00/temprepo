# الطلبان 7 و 8 — التحكم في الأقفال (Concurrency Control) وسلامة المعاملات (ACID)

> ملف توثيقي لحل الطلبين 7 و8 في مشروع **High-Performance E-Commerce Backend Engine**.
> شرح تفصيلي، ثم توضيح الفرق عن الطلب الأول، ثم شرح مبسّط في النهاية.

كل المنطق في: `app/Services/CheckoutService.php`

---

## 1. ما الذي يطلبه كل متطلب

| المتطلب | النص | المطلوب عملياً |
|---------|------|----------------|
| **الطلب 7** — Concurrency Control | تطبيق قفل متفائل (Optimistic) أو تشاؤمي (Pessimistic) عند تحديث المخزون الحسّاس | **قفل موزّع (Redis)** — حسب توجيه الأستاذ: في مكان غير أقفال قاعدة البيانات |
| **الطلب 8** — Transaction Integrity / ACID | ضمان أن العمليات المركّبة (تحديث مخزون + إنشاء طلب) تنجح كلها أو تفشل كلها، حتى تحت الوصول المتزامن | **`DB::transaction`** تغلّف كل العمليات |

---

## 2. حل الطلب الثامن (ACID) — `DB::transaction`

كل العمليات المركّبة داخل معاملة واحدة:

```php
$order = DB::transaction(function () use (...) {
    // 1) خصم المخزون لكل منتج
    // 2) إنشاء الطلب (Order)
    // 3) إنشاء بنود الطلب (OrderItems)
    // 4) تفريغ سلة المستخدم
    return $order;
});
```

### كيف تتحقق خصائص ACID

| الخاصية | التحقق |
|---------|--------|
| **A**tomicity (الذرّية) | أي استثناء داخل المعاملة → `rollback` تلقائي يُرجع **كل** التغييرات (لا طلب نصف مكتمل) |
| **C**onsistency (الاتساق) | شرط `stock >= qty` يمنع المخزون السالب؛ الطلب لا يُنشأ إلا بعد نجاح كل الخصومات |
| **I**solation (العزل) | الخصم الذري المشروط + القفل الموزّع يمنعان تداخل الطلبات المتزامنة |
| **D**urability (الديمومة) | بعد `commit` تُحفظ البيانات في القرص دائماً |

**مثال على الذرّية:** لو نجح خصم المنتج الأول ثم فشل الثاني (نفد مخزونه) → يُرمى استثناء → **يُلغى خصم الأول أيضاً** ولا يُنشأ أي طلب. لا يوجد "نصف عملية".

## 3. حل الطلب السابع (Distributed Lock) — `Cache::lock`

استخدمنا قفل Laravel الذرّي المبني على **Redis** (`Cache::lock`)، وهو قفل موزّع حقيقي (يعتمد `SET NX` في Redis)، منفصل تماماً عن أقفال MySQL.

```php
// IDs مرتّبة → كل الطلبات تقفل بنفس الترتيب → لا deadlock عند تشارك المنتجات
$productIds = $cartItems->pluck('product_id')->unique()->sort()->values();

$locks = [];
foreach ($productIds as $lockProductId) {
    $lock = Cache::lock("lock:product:{$lockProductId}", 10);   // TTL أمان 10 ثوانٍ

    if (! $lock->block(5)) {                                     // انتظر حتى 5 ثوانٍ
        foreach ($locks as $acquired) { $acquired->release(); }  // أطلق ما حُجز
        throw new RuntimeException("System busy, please retry ...");
    }
    $locks[] = $lock;
}

try {
    $order = DB::transaction(function () { /* خصم + طلب (Req 8) */ });
} finally {
    foreach ($locks as $lock) { $lock->release(); }              // أطلق دائماً
}
```

### قرارات التصميم ولماذا

- **قفل لكل منتج** (`lock:product:{id}`): يُسلسِل فقط الطلبات التي تلمس **نفس** المنتج؛ منتجات مختلفة تُعالج بالتوازي (لا نخنق الأداء).
- **ترتيب الـ IDs قبل القفل**: يضمن أن كل الطلبات تحجز الأقفال بنفس الترتيب → **يمنع الـ deadlock** عند وجود منتجات مشتركة بين سلّتين.
- **`->block(5)`**: ينتظر حتى 5 ثوانٍ للحصول على القفل بدل الفشل الفوري.
- **TTL = 10 ثوانٍ**: شبكة أمان — لو مات العامل (process) قبل `release`، يُحرّر القفل تلقائياً فلا يبقى عالقاً للأبد.
- **`finally`**: يضمن تحرير الأقفال حتى لو رمت المعاملة استثناءً.

### لماذا قفل موزّع تحديداً (وليس قفل DB أو in-process)؟

```
قفل in-process (mutex)  → يعمل داخل عملية PHP واحدة فقط ❌ (Apache يشغّل عمليات متعددة)
قفل DB-row (FOR UPDATE) → يعمل، لكنه على مستوى قاعدة البيانات (ما طلبه الأستاذ تجنّبه)
قفل Redis (Cache::lock) → مشترك عبر كل عمليات وخوادم التطبيق ✅
```

هذا يرتبط مباشرة بالطلب 5 (موزّع الأحمال): عندما يعمل التطبيق على عدة خوادم خلف Load Balancer، فإن قفل Redis هو الوحيد الذي يضمن أن خادمين مختلفين لا يبيعان نفس القطعة في آنٍ واحد.

## 4. سؤال مهم: هل الخصم الذري قفل تشاؤمي على مستوى قاعدة البيانات؟

**لا.** هذا تمييز دقيق ومهم:

```sql
UPDATE products SET stock = stock - :qty WHERE id = :id AND stock >= :qty
```

- هذا **تحديث ذرّي مشروط بلا قفل** (lock-free atomic conditional update / أسلوب Compare-And-Set). الشرط (`stock >= qty`) والكتابة يحدثان **في عبارة واحدة غير قابلة للتجزئة**.
- فلسفته أقرب إلى **المتفائل (Optimistic)**: لا يُمسك قفلاً صريحاً، بل يعتمد على شرط، ويفشل بأمان (`$updated == 0`) إن لم يتحقق الشرط.
- صحيح أن InnoDB يأخذ **قفل صف داخلياً** للحظة تنفيذ هذه العبارة الواحدة — لكن هذا سلوك تلقائي لأي `UPDATE`، وليس **نمط القفل التشاؤمي**.

**القفل التشاؤمي الحقيقي على مستوى DB** مختلف — هو أن تمسك القفل صراحةً وتحتفظ به عبر قراءة-ثم-كتابة:
```php
$product = Product::where('id', $id)->lockForUpdate()->first();  // SELECT ... FOR UPDATE
// ... منطق ...
$product->decrement('stock', $qty);                              // القفل محتجز طوال هذه المدة
```

**الخلاصة:**

| الأسلوب | التصنيف | يُمسك قفلاً صريحاً؟ |
|---------|---------|----------------------|
| `WHERE stock >= qty ->decrement` | ذرّي مشروط بلا قفل (≈ متفائل) | لا (قفل صف داخلي لحظي تلقائي فقط) |
| `lockForUpdate()` | تشاؤمي على مستوى DB | نعم (صريح، محتجز عبر العملية) |
| `Cache::lock()` (Redis) | تشاؤمي **موزّع** (مستوى التطبيق) | نعم (صريح، عبر كل الخوادم) |

## 5. الفرق بين الطلب الأول والطلبين 7 و 8

الثلاثة تتعامل مع التزامن لكن **كلٌّ يركّز على زاوية مختلفة**، وهم **يتراكبون طبقات**:

```
الطلب 7  →  قفل موزّع (Redis)          ← الطبقة الخارجية: إقصاء متبادل عبر الخوادم
              │
الطلب 8  →    DB::transaction (ACID)     ← الطبقة الوسطى: الكل-أو-لا-شيء
                │
الطلب 1  →      الخصم الذري المشروط       ← الطبقة الداخلية: منع overselling
```

| | الطلب 1 | الطلب 7 | الطلب 8 |
|--|---------|---------|---------|
| **التركيز** | المشكلة (Race Condition) وإثبات حلّها | تقنية القفل (موزّع) | غلاف المعاملة (ACID) |
| **التسليم** | مسار غير آمن للمقارنة + الخصم الذري كحل | قفل Redis موزّع | `DB::transaction` |
| **المستوى** | عبارة DB واحدة ذرّية | طبقة التطبيق / Redis | معاملة DB |
| **يجيب على** | "كيف نمنع بيع قطعتين من واحدة؟" | "كيف نُسلسِل الوصول للمورد الحسّاس عبر الخوادم؟" | "كيف نضمن أن العملية المركّبة لا تكتمل نصفياً؟" |

- **الطلب 1** هو القلب: المشكلة (race) وحلّها الذرّي. وحده يكفي لمنع الـ overselling.
- **الطلب 8** يغلّف الخصم مع إنشاء الطلب وبنوده وتفريغ السلة في وحدة واحدة لا تتجزأ.
- **الطلب 7** يضيف إقصاءً متبادلاً صريحاً عبر الخوادم (دفاع في العمق + شرط الأستاذ + قيمة في بيئة الطلب 5).

## 6. كيف تُختبر

### الطلب 8 (ACID — rollback)
اختبار يفرض فشلاً في منتصف المعاملة (مثلاً منتج ثانٍ نفد) ويتحقّق:
- لم يُنشأ أي `Order`.
- مخزون المنتج الأول **رجع كما كان** (أُلغي خصمه).

### الطلب 7 (القفل الموزّع — تزامن)
محاكاة تزامن مكثّف عبر **عمليات منفصلة** (k6 / JMeter)، لأن القفل لا يظهر أثره داخل عملية PHP واحدة:
```
5 طلبات متوازية على منتج مخزونه 1  →  ينجح طلب واحد فقط، وتُرفض الأربعة بأمان
```
> ملاحظة: الخصم الذري وحده يحقق هذه النتيجة أيضاً؛ القفل الموزّع يجعل التسلسل **صريحاً** ويمدّه ليعمل عبر عدة خوادم (الطلب 5).

# الشرح المبسّط 🟢

تخيّل متجراً فيه **قطعة واحدة** من منتج، ودخل **5 زبائن** يريدونها كلهم في نفس الثانية.

### الطلب 8 (المعاملة) — "الكل أو لا شيء"
عملية الشراء = عدة خطوات (أنقص المخزون، أنشئ الفاتورة، فرّغ السلة). الطلب 8 يقول:
**نفّذها كحزمة واحدة** — إن فشلت خطوة، تراجع عن كل الخطوات. لا توجد "فاتورة بنصف بضاعة".

### الطلب 1 (الخصم الذري) — "تحقّق واخصم بحركة واحدة"
بدل أن نسأل "هل يوجد مخزون؟" ثم نخصم (وبينهما قد يسبقنا غيرنا)، ندمجهما في **أمر واحد**:
"إن كان المخزون ≥ المطلوب، اخصمه الآن". لا توجد لحظة بينهما يتسلّل فيها زبون آخر.

### الطلب 7 (القفل الموزّع) — "مفتاح الغرفة"
نضع **مفتاحاً واحداً** على باب كل منتج في Redis. من يمسك المفتاح يدخل ويشتري، والبقية ينتظرون
دورهم. وبما أن المفتاح في Redis (مكان مشترك)، فهو يعمل **حتى لو كان المتجر موزّعاً على عدة فروع
(خوادم)** — لا يستطيع فرعان بيع نفس القطعة معاً.

### لماذا ثلاث طبقات؟
- **الخصم الذري (1)** وحده يمنع البيع الزائد.
- **المعاملة (8)** تضمن أن باقي خطوات الشراء لا تكتمل نصفياً.
- **القفل الموزّع (7)** يضيف تنظيماً صريحاً يعمل عبر كل الخوادم.

### الفرق بسطر واحد
> الطلب 1 = "تحقّق واخصم بحركة واحدة". الطلب 8 = "نفّذ كل خطوات الشراء كحزمة أو ألغِها كلها".
> الطلب 7 = "مفتاح مشترك في Redis يمنع أي خادمين من لمس نفس المنتج معاً".

### وهل الخصم الذري قفل؟
**لا** — إنه ليس قفلاً تُمسكه وتنتظر. إنه "تحقّق + خصم" في خطوة واحدة لا تتجزأ. القفل الحقيقي
(تمسكه وتحجزه) موجود في الطلب 7 عبر Redis.

# شرح إصلاح مشكلة Race Condition في Checkout

## ملخص المشكلة

كانت عملية checkout تحتوي على ثغرة تزامن (Race Condition) عند خصم المخزون.

قبل الإصلاح، التنفيذ كان كالتالي:

1. قراءة قيمة المخزون الحالية.
2. التحقق أن المخزون يكفي.
3. إنقاص الكمية.
4. حفظ المنتج.

عند وصول طلبين متزامنين، قد يقرأ الطلبان نفس قيمة المخزون ويُكملان بنجاح، فينتج بيع أكثر من الكمية المتاحة.

---

## السبب الجذري

المنطق السابق كان يعتمد على `read then write` بدون حجز ذري للمخزون داخل قاعدة البيانات.
هذا النمط غير آمن مع التزامن العالي.

سبب الخطر هو أن التحقق من الكمية كان يتم في التطبيق قبل التحديث الفعلي، لذلك يمكن لطلبين قراءة نفس القيمة قبل أن يكتب أي منهما النتيجة النهائية.

---

## ما تم تطبيقه فعليًا

### 1) حجز المخزون بشكل ذري داخل Transaction

تم تعديل [app/Services/CheckoutService.php](app/Services/CheckoutService.php) لاستخدام:

- `DB::transaction(...)`
- خصم ذري لكل عنصر سلة عبر:
  - `where('stock', '>=', $qty)->decrement('stock', $qty)`

إذا كان ناتج التحديث `0` فهذا يعني أن المخزون غير كافٍ لحظة التنفيذ، فيفشل checkout بشكل آمن ويتم التراجع عن المعاملة.

لماذا هذا آمن؟

- لأن شرط `WHERE stock >= qty` يتم تقييمه داخل قاعدة البيانات لحظة تنفيذ عملية التحديث نفسها، وليس مسبقًا داخل التطبيق.
- هذا يلغي نافذة السباق بين التحقق والتحديث، ويمنع تجاوز المخزون (overselling).

### 2) منع التعديل المتزامن على السلة لنفس المستخدم

داخل نفس المعاملة، يتم جلب عناصر السلة باستخدام `lockForUpdate()`.

الفائدة:

- يتم منع التعديل المتزامن على عناصر السلة لنفس المستخدم أثناء تنفيذ عملية checkout.

### 3) إنشاء الطلب بطريقة متوافقة مع مخطط المشروع

المشروع الحالي يخزن تفاصيل المنتجات في `order_items` وليس في `orders`.

التدفق المطبق:

1. إنشاء طلب بحالة `pending`.
2. إنشاء عناصر الطلب من عناصر السلة.
3. حذف عناصر السلة التي تمت معالجتها.

### 4) إضافة خدمة دفع وهمية

تمت إضافة [app/Services/PaymentService.php](app/Services/PaymentService.php).

السلوك الحالي:

- ترجع نجاحًا تجريبيًا (`success`) بدون أي تكامل مع API خارجي.

### 5) تأكيد الطلب أو التعويض

بعد الحجز وإنشاء الطلب:

- عند نجاح الدفع: يتم تحديث الحالة إلى `paid`.
- عند فشل الدفع أو حدوث Exception:
  1. تحديث حالة الطلب إلى `failed`.
  2. إعادة الكميات للمخزون لكل عنصر طلب باستخدام `increment(...)` داخل Transaction تعويضية.

هذا يضمن عدم بقاء النظام في حالة غير متسقة بين الطلب والمخزون.

ملاحظة هندسية متقدمة:

- التعويض اليدوي مناسب جدًا لهذا النطاق الأكاديمي.
- في بيئات الإنتاج الكبيرة، غالبًا يتم دعم ذلك عبر حالات طلب واضحة + Events/Jobs قابلة لإعادة المحاولة (retry) لضمان التعافي حتى مع فشل الخدمات الخارجية.

## النتيجة بعد الإصلاح

- لم يعد بالإمكان تجاوز المخزون بسبب نافذة التزامن القديمة.
- checkout إمّا ينجح بحجز آمن، أو يفشل برسالة نفاد مخزون.
- فشل الدفع لا يترك الطلب أو المخزون في حالة غير صحيحة.
- التصميم يعتمد أساسًا على عمليات ذرية داخل قاعدة البيانات (database-level atomic operations) بدل الاعتماد على أقفال على مستوى التطبيق (application-level locking)، وهذا يقلل التضارب ويحسن السلوك تحت الضغط.

## تحسينات مستقبلية مقترحة

1. إضافة `Idempotency Key` لطلب checkout لمنع تكرار التنفيذ عند إعادة الإرسال.
2. تسجيل أدق لخطوات الدفع (Audit Trail) مع حالات retry.
3. نقل معالجة الدفع إلى Queue/Job لسيناريوهات الإنتاج.
4. إضافة Sequence Diagram لتدفق checkout (حجز مخزون -> إنشاء طلب pending -> دفع -> paid/failed -> تعويض عند الفشل) لتسهيل المناقشة.

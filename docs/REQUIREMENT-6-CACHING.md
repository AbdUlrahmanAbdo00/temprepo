# الطلب السادس — التخزين المؤقت الموزّع (Distributed Caching)

> ملف توثيقي لحل الطلب السادس في مشروع **High-Performance E-Commerce Backend Engine**.
> يحتوي على شرح تفصيلي ثم شرح مبسّط في النهاية.

---

## 1. ما الذي يطلبه المتطلب

نص الطلب السادس:

> **Distributed Caching:** دمج طبقة تخزين مؤقت (مثل Redis) لتخزين المنتجات الأكثر طلباً
> وتقليل الاستعلامات المباشرة من قاعدة البيانات.

أي ثلاثة أهداف:

1. طبقة تخزين مؤقت حقيقية (Redis).
2. الاحتفاظ بالبيانات الأكثر طلباً قريبة وسريعة.
3. تقليل عدد الاستعلامات المباشرة على قاعدة البيانات.

---

## 2. النمط المعتمد: Cache-Aside (Lazy Loading)

اعتمدنا نمط **Cache-Aside** — وهو النمط القياسي للقراءة:

```
طلب البيانات
   │
   ├─ موجودة في Redis؟  ── نعم ──▶ أرجعها من الكاش (بلا DB)   ← Cache HIT
   │
   └─ لا ──▶ اجلبها من DB ──▶ خزّنها في Redis ──▶ أرجعها       ← Cache MISS
```

**لماذا هذا النمط:**
- التطبيق هو المتحكّم: يخزّن فقط ما يُطلب فعلاً.
- "الأكثر طلباً" يتحقق **ضمنياً**: المنتج المطلوب كثيراً يبقى دافئاً في الكاش (نسبة إصابة عالية)،
  والمنتج النادر يسقط بانتهاء صلاحيته. لا حاجة لجدول إحصائيات.
- قاعدة البيانات تبقى **مصدر الحقيقة** دائماً؛ الكاش مجرد طبقة تسريع.

## 3. التطبيق في الكود

### 3.1 قائمة المنتجات — `GET /api/products`

`app/Http/Controllers/Api/ProductController.php`

```php
public function index(): JsonResponse
{
    // Pagination لتصغير حجم الاستجابة (يعالج اختناق الحمولة — الطلب 10)
    // كل صفحة تُخزّن بمفتاح مستقل مع TTL = 5 دقائق (الطلب 6)
    $page = request()->integer('page', 1);

    $products = Cache::remember("products:page:{$page}", now()->addMinutes(5), function () {
        return Product::query()->latest()->paginate(20);
    });

    return response()->json($products);
}
```

- المفتاح: `products:page:1`, `products:page:2`, ... (كل صفحة مستقلة).
- TTL = 5 دقائق → بعدها يُعاد الجلب من DB تلقائياً ويُخزّن من جديد.

### 3.2 منتج واحد — `GET /api/products/{id}`

```php
public function show(int $id): JsonResponse
{
    // استعلام واحد فقط على مسار الـ Cache MISS — نُبقي المسار خفيفاً
    $product = Cache::remember("product:{$id}", now()->addMinutes(10), function () use ($id) {
        return Product::findOrFail($id);
    });

    return response()->json(['data' => $product]);
}
```

- المفتاح: `product:5` لكل منتج على حدة.
- TTL = 10 دقائق.
- **استعلام واحد فقط** عند الـ MISS (لا تتبّع مشاهدات ولا أي عمل إضافي يثقل DB).

### 3.3 إبطال الكاش عند البيع — `app/Services/CheckoutService.php`

```php
// بعد نجاح المعاملة (Req 6) — يُبطل كاش المنتج الفردي فقط
foreach ($cartItems as $cartItem) {
    Cache::forget("product:{$cartItem->product_id}");
}
```

- يُبطَل `product:{id}` للمنتجات المُباعة → الطلب التالي يقرأ المخزون المحدّث من DB.
- **القائمة `products:page:*` لا تُبطَل نشطاً** — تعتمد على TTL القصير (≤5 دقائق).
- يُنفّذ **بعد** نجاح الـ transaction، لأن `Cache::forget` لا يمكنه المشاركة في معاملة DB.

## 4. لماذا القائمة تعتمد على TTL ولا تُبطَل عند البيع؟ (ربط بالطلب 1)

```
القائمة (products:page:N)  =  للعرض فقط
قرار البيع الحقيقي        =  في CheckoutService على DB مباشرة (الخصم الذري المشروط)
```

```php
// CheckoutService — القرار الحقيقي للمخزون يحدث هنا، لا في الكاش
Product::where('id', $productId)
    ->where('stock', '>=', $qty)
    ->decrement('stock', $qty);   // ذرّي — يرفض إن لم يكن المخزون كافياً
```

لذلك حتى لو عرضت القائمة "متوفر" لثوانٍ بعد نفاد المنتج، فإن لحظة الشراء **الخصم الذري يرفض**
ولا يحدث **overselling**. هذا يجعل إبطال القائمة النشط تعقيداً غير ضروري.

---

## 5. حدّ الذاكرة — أين يوضع؟ (للتقرير)

السؤال: "ماذا لو امتلأت ذاكرة Redis؟" — الجواب **ليس** حدّاً في كود التطبيق، بل في إعداد Redis:

```conf
# redis.conf
maxmemory 256mb
maxmemory-policy allkeys-lfu   # يحتفظ بالأكثر طلباً، يطرد الأقل طلباً تلقائياً
```

- `allkeys-lfu` = Least Frequently Used → الطرد التلقائي للأقل طلباً.
- عند حجم المشروع (آلاف المنتجات ≈ بضعة MB) **لا تُفعَّل أبداً** — كل شيء يتّسع.
- الطرد لا يُفسد البيانات: أي مفتاح غير موجود يُجلب من DB. أسوأ نتيجة = طلب أبطأ، لا بيانات خاطئة.

## 6. قرار مهم: لماذا حُذف جدول `product_statistics`

في نسخة سابقة أُضيف جدول `product_statistics` (purchase_count / view_count) لترتيب
"الأكثر طلباً" صراحةً. **حُذف بالكامل** للأسباب التالية:

- الطلب السادس يطلب **تقليل استعلامات DB**، لا الترتيب حسب المبيعات.
- تتبّع المشاهدات داخل `show()` كان يضيف استعلامين إضافيين على كل **Cache MISS**
  (`SELECT` + `INSERT/UPDATE`)، فيضاعف الحمل على DB — وهو عكس هدف المتطلب تماماً.
  (تحت الضغط بـ 3210 منتجاً = آلاف الاستعلامات الزائدة.)
- "الأكثر طلباً" يتحقق ضمنياً عبر نسبة الإصابة + سياسة `allkeys-lfu` في Redis نفسه.

**الدرس:** لا تُضِف تعقيداً لا يطلبه المتطلب. البساطة هنا هي الإجابة الهندسية الصحيحة.


# الشرح المبسّط 🟢

تخيّل أن قاعدة البيانات **مستودع كبير بعيد**، و Redis **رفّ صغير سريع بجانب الموظّف**.

### القراءة
عندما يطلب زبون منتجاً:
1. الموظّف ينظر إلى الرفّ القريب (Redis) أولاً.
2. **موجود؟** يعطيه فوراً. ⚡ (سريع — بلا مشوار للمستودع)
3. **غير موجود؟** يذهب للمستودع البعيد (DB)، يحضره، **ويضع نسخة على الرفّ** قبل أن يعطيه.
   فالزبون التالي الذي يطلب نفس المنتج يجده على الرفّ جاهزاً.

النتيجة: المنتجات المطلوبة كثيراً تبقى دائماً على الرفّ القريب → خدمة أسرع وضغط أقل على المستودع.

### النسيان التلقائي (TTL)
كل نسخة على الرفّ لها "تاريخ صلاحية" (5 أو 10 دقائق). بعدها تُرمى تلقائياً،
فإذا طُلبت ثانية تُحضَر طازجة من المستودع. هذا يمنع عرض بيانات قديمة جداً.

### المسح عند البيع (forget)
عندما يُباع منتج ويتغيّر مخزونه، الموظّف **يمزّق نسخة ذلك المنتج من الرفّ فوراً**،
حتى لا يعطي أحداً معلومة مخزون قديمة. الطلب التالي يحضر النسخة المحدّثة.

### الصفحات (Pagination)
بدل أن يحمل الموظّف **كل** 3210 منتج دفعة واحدة (ثقيل وبطيء)، يحضر **20 منتجاً في الصفحة**.
أخفّ بكثير وأسرع في الإرسال.

### لماذا لا نخاف من بيع منتج نفد؟
الرفّ (الكاش) **للعرض فقط**. لحظة الدفع الحقيقية، الموظّف يتحقق من **المستودع نفسه** (DB)
بعملية واحدة لا تتجزأ: "هل يوجد كمية كافية؟ إن نعم اخصم، وإلا ارفض". لذلك حتى لو أظهر الرفّ
"متوفر" خطأً لثوانٍ، البيع الفعلي يُرفض إن لم تكن البضاعة موجودة → **لا بيع زائد**.

### الخلاصة بسطر واحد
> نخزّن المطلوب على رفّ سريع (Redis)، ننساه بعد دقائق تلقائياً، ونمسحه فوراً عند تغيّره —
> فنخدم العملاء أسرع ونريح قاعدة البيانات، دون أي بيانات خاطئة.

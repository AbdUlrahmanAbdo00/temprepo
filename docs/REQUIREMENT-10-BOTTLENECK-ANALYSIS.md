# الطلب العاشر — القياس وتحديد الاختناقات (Benchmarking & Bottleneck Analysis)

> ملف توثيقي لحل الطلب العاشر في مشروع **High-Performance E-Commerce Backend Engine**.
> شرح تفصيلي، ثم شرح مبسّط. كل الأرقام من قياسات حقيقية.

---

## 1. ما الذي يطلبه المتطلب

> قياس زمن الاستجابة لأهمّ العمليات، تحديد **اختناق واحد على الأقل**، وتقديم **مقارنة رقمية قبل/بعد** التحسين.

---

## 2. منهجية القياس (قِس، لا تخمّن)

أضفنا **تجهيزات قياس (instrumentation)** بلا تغيير أي منطق:

| الأداة | تقيس |
|--------|------|
| `MeasureProcessingTime` middleware | زمن المعالجة داخل PHP (`X-Process-Time-Ms`) |
| `ProfileDbQueries` middleware | عدد استعلامات SQL + مجموع زمنها + أبطأها |
| `CheckoutService::$timings` | زمن كل مرحلة داخل الشراء (قفل، خصم، إنشاء، commit، dispatch، log، load) |
| ترويسة `X-Checkout-Profile` | تُمرّر التفكيك إلى k6 لحساب p95/avg لكل مرحلة |

**التمييز المفتاحي:** الزمن الكلّي (يقيسه k6) مقابل زمن المعالجة داخل PHP. الفرق بينهما = **انتظار خيط Apache** (الطلب ينتظر دوره قبل أن يبدأ التنفيذ).

---

## 3. تفكيك زمن الشراء (القياس الحقيقي)

```
الكلّي p95            ≈ 14 ثانية
المعالجة داخل PHP     ≈ 6.5s p95 (avg ~5s)
انتظار الباب (الفرق)  ≈ 7.5s   ← خيوط محتجَزة بالعمل البطيء
```

### تفكيك المعالجة داخل PHP (% من avg، مرتّب)

| المرحلة | % | ملاحظة |
|---------|---|--------|
| **إطار العمل + auth (Sanctum) + تسلسل JSON** | **~48%** | متبقٍّ غير مقيس مباشرة؛ يتضخّم بالتزامن |
| **dispatch إلى Redis (مهمتان)** | **~23%** | أكبر عملية مفردة (predis تحت التزامن) |
| **استعلامات SQL (13 استعلاماً)** | **~20%** | مجموع ~1100ms، لا N+1 |
| التسجيل (3× Log::info) | ~7% | كتابة ملف log واحد تحت التزامن |
| commit (fsync) | ~3.6% | المعاملة نفسها صغيرة |
| قفل Redis (الطلب 7) | ~0.8% | ضئيل — ليس عنقاً |

**أكبر اختناق مُقاس:** ليس قاعدة البيانات ولا الأقفال — بل **dispatch الطوابير (~23%) + حِمل إطار العمل/التسلسل (~48%)**، وكلّه **متضخّم بإشباع الجهاز الواحد** (المعالج 29% فقط = ليس حوسبة، بل تنازع/انتظار).

---

## 4. المقارنات الرقمية قبل/بعد (تجارب فعلية)

### 4.1 Pagination — اختناق الحمولة الضخمة
```
قبل: GET /api/products يُرجع كل المنتجات → ~38 MB لكل طلب → p95 ~9.4s
بعد: Pagination (20/صفحة) → ~1.5 MB → حمولة خفيفة
```

### 4.2 التخزين المؤقت (الطلب 6) — تقليل استعلامات DB
```
بدون كاش: 150 استعلام / 50 طلب
مع Redis: 7 استعلامات → تقليل ~95%
```

### 4.3 محرّك الطوابير DB → Redis — استهدف dispatch
```
QUEUE=database: المعالجة داخل PHP = 9.6s p95  (كل dispatch = INSERT+fsync في MySQL)
QUEUE=redis:    المعالجة داخل PHP = 6.3s p95  → −34%
بلا أي مساس بـ Durability (الطلب 8) — أنظف من تعديل MySQL
```

### 4.4 config:cache — أصلح أخطاء التزامن
```
قبل: تحت 100 متزامن، تنازع قراءة .env → forge / Class "Redis" not found → 5xx
بعد config:cache: لا قراءة .env لكل طلب → أخطاء الخادم = 0
```

### 4.5 رفع خيوط Apache 150 → 300 — تجربة نفي
```
150 thread: p95 ~14-17s | in-PHP ~6-8s
300 thread: p95 ~13.9s  | in-PHP ~6.5s   → لا فرق
الاستنتاج: الخيوط ليست العنق (مُثبَت تجريبياً). 150 > 100 أصلاً؛
الخيوط محتجَزة بالعمل البطيء، فزيادتها = طلبات أكثر تقصف نفس MySQL/Redis.
```

### 4.6 التوزيع على جهاز واحد (الطلب 5) — تجربة نفي معماري
```
مباشر (:8000):        p95 ~14s
عبر الموزّع (8000+8080 artisan أحادي الخيط):  p95 ~40.8s  → أسوأ 2.5×!
الاستنتاج: التوزيع على جهاز واحد مع خادم ثانٍ ضعيف يسحب النظام للأسفل.
التوسّع الأفقي يفيد فقط مع خوادم متكافئة على عتاد منفصل.
```

---

## 5. الاختناق المُحدَّد والاستنتاج المعماري

```
الاختناق الرئيسي = سقف إنتاجية الجهاز الواحد تحت الحمل المتزامن،
موزّع على: dispatch الطوابير + استعلامات SQL + حِمل إطار العمل/التسلسل.
(ليس الأقفال 0.8%، ولا الخيوط، ولا الـ commit وحده 3.6%.)

الحل المعماري الحقيقي:
  • توسّع أفقي على عتاد منفصل (خوادم تطبيق متكافئة)
  • قاعدة بيانات أقوى/منفصلة (MySQL هو المورد المشترك)
  • نقل العمل الثقيل للطوابير (الطلب 3) — مطبَّق
```

---

## 6. ملاحظات منهجية صادقة

- **النسب من المتوسطات** (قابلة للجمع)؛ الـ p95 غير قابل للجمع.
- **كل الأرقام متضخّمة بالتزامن**: نفس العمليات بالميلي ثانية عند الخمول، وبالثواني تحت 100 متزامن. أي أن جزءاً كبيراً من "الـ48%" هو انتظار المعالج (descheduling)، لا عملية بطيئة خوارزمياً.
- البيئة محلية (XAMPP/جهاز واحد) → النتائج صاخبة؛ تُقرأ كاتجاهات لا أرقام مطلقة.

# الشرح المبسّط 🟢

سؤال الطلب 10: **"عملية الشراء تأخذ ~14 ثانية تحت الضغط. أين تذهب هذه الثواني؟"**

### كيف عرفنا؟
بدل التخمين، وضعنا "ساعة توقيت" حول كل خطوة في الشراء، وعدّاداً لاستعلامات قاعدة البيانات. ثم شغّلنا الضغط وقرأنا الأرقام.

### أين تذهب الثواني؟
```
نصفها تقريباً : إطار العمل + تجهيز الرد (يتضخّم لأن الجهاز مزدحم)
الربع        : إرسال المهام لـ Redis (الطوابير)
الخُمس        : استعلامات قاعدة البيانات (13 استعلاماً)
الباقي صغير  : تسجيل، حفظ، قفل
```

النتيجة المفاجئة: **قاعدة البيانات والأقفال ليست المشكلة الكبرى** — بل ازدحام الجهاز الواحد تحت 100 طلب معاً، موزّعاً على إرسال الطوابير وتجهيز الردود.

### ماذا جرّبنا (قبل/بعد)؟
- **صفحات بدل القائمة كاملة:** 38 ميغابايت → 1.5 ميغابايت لكل طلب.
- **كاش Redis:** استعلامات DB ↓95%.
- **طوابير على Redis بدل DB:** زمن المعالجة ↓34%.
- **config:cache:** أخطاء التزامن → صفر.
- **مضاعفة الخيوط (150→300):** لا فرق! (أثبتنا أن الخيوط ليست السبب.)
- **توزيع على جهاز واحد:** صار أسوأ! (الخادم الثاني الضعيف عطّل الكل.)

### الخلاصة بسطر واحد
> قِسنا فوجدنا الاختناق = سقف الجهاز الواحد (إرسال الطوابير + الاستعلامات + الإطار)، لا الأقفال ولا الخيوط. الحل الحقيقي للآلاف: توزيع على أجهزة منفصلة + قاعدة بيانات أقوى.

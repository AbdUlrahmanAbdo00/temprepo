# شرح المشروع والتعديل الأخير

## نبذة عن المشروع

هذا المشروع عبارة عن متجر إلكتروني مبني باستخدام Laravel، ويهدف إلى عرض الأساسيات الأساسية لعملية البيع عبر API بشكل منظم وبسيط. المشروع يحتوي على:

- تسجيل الدخول والتسجيل باستخدام Laravel Sanctum.
- عرض المنتجات مع السعر والمخزون والوصف.
- سلة مشتريات مرتبطة بالمستخدم.
- إنشاء الطلبات وعناصر الطلب.
- خدمة مخصصة لإدارة عملية الـ checkout.

## الفكرة العامة

الهدف من المشروع هو بناء مسار شراء واضح يبدأ من إضافة المنتج إلى السلة، ثم تنفيذ عملية الشراء، ثم إنشاء الطلب وتحديث حالة الطلب. التركيز الأساسي كان على جعل البنية واضحة وقابلة للتطوير لاحقًا، وليس فقط على الواجهة أو العرض.

## التعديل الأخير

التعديل الأخير يركز على تحسين checkout من ناحية المخزون والدفع التجريبي بدون الحاجة إلى API دفع خارجي.

### 1. حجز المخزون بشكل Atomic

بدل الاعتماد على قراءة المخزون ثم تعديله لاحقًا، يتم تحديثه من خلال Query واحدة داخل Transaction:

```php
DB::transaction(function () use ($productId, $qty) {
    $updated = Product::where('id', $productId)
        ->where('stock', '>=', $qty)
        ->decrement('stock', $qty);

    if (! $updated) {
        throw new Exception('Out of stock');
    }
});
```

هذا الأسلوب يمنع مشكلة Race Condition لأن قاعدة البيانات نفسها تتحقق من الشرط وتنفذ التعديل في نفس الوقت.

### 2. خدمة دفع وهمية

بما أن المشروع جامعي ولا يحتاج دمج API دفع حقيقي، يتم استخدام خدمة دفع وهمية مثل:

```php
class PaymentService
{
    public function pay($order)
    {
        return ['status' => 'success'];
    }
}
```

هذا يحقق متطلب الدفع بشكل عملي داخل المشروع بدون الاعتماد على مزود خارجي.

### 3. تأكيد الطلب أو إرجاع المخزون

بعد نجاح الدفع يتم تحديث الطلب إلى `paid`، وإذا فشل الدفع يتم تحويله إلى `failed` ثم إرجاع المخزون.

## لماذا هذا التعديل مهم

- يحمي المخزون من التعارض عند تعدد الطلبات.
- يجعل checkout أكثر استقرارًا تحت الضغط.
- يلغي الحاجة إلى API دفع خارجي في المشروع الجامعي.
- يحافظ على إمكانية التطوير لاحقًا إذا أردت استبدال الدفع الوهمي بخدمة حقيقية.

## ما تم تطبيقه فعليًا في الكود

- تم تحديث [app/Services/CheckoutService.php](../app/Services/CheckoutService.php) ليستخدم `DB::transaction` مع خصم مخزون ذري عبر Query مباشرة.
- تم إضافة [app/Services/PaymentService.php](../app/Services/PaymentService.php) كخدمة دفع وهمية ترجع نجاحًا تجريبيًا.
- أصبح checkout يغيّر حالة الطلب إلى `paid` عند نجاح الدفع، أو `failed` عند فشله، مع الحفاظ على اتساق المخزون.

## خلاصة

المشروع الآن يوضح بشكل أفضل كيف تعمل دورة الشراء في متجر إلكتروني: سلة، طلب، خصم مخزون، ثم دفع تجريبي، ثم تأكيد أو إلغاء. هذا يجعل المشروع مناسبًا للشرح الأكاديمي وللعرض العملي أيضًا.

# حل المتطلب الثاني: Resource Management & Capacity Control

هذا الملف يشرح أبسط تطبيق مناسب للمتطلب الثاني في مشروع **High-Performance E-Commerce Backend Engine** دون تكبير المشروع أو إدخال تعقيد غير ضروري.

## الفكرة المعتمدة

أفضل تطبيق حاليًا على هذا المشروع هو:

- Redis لإدارة الـ cache والـ queue.
- عدد محدود من Queue Workers بدل التشغيل غير المحدود.
- Rate limiting على المسارات الحساسة.
- تأجيل المهام غير الحرجة إلى الخلفية.

هذا يناسب مشروع Laravel الموجود لأن البنية الحالية تعتمد أصلًا على خدمة checkout منفصلة، ومبدئيًا يمكن ضبط التحسينات على مستوى الإعدادات والـ routes دون إعادة بناء النظام.

## ماذا يطبق الآن

### 1) Redis

يُستخدم Redis لتقليل الضغط على قاعدة البيانات وتسريع الوصول إلى البيانات المؤقتة والمهام الخلفية.

الملفات المرتبطة:

- `config/queue.php`
- `config/cache.php`
- `.env`

الإعداد المقترح:

```env
QUEUE_CONNECTION=redis
CACHE_DRIVER=redis
```

### 2) Queue Workers محدودة

بدل تشغيل عدد غير محدود من العمال، يتم تشغيل عدد ثابت ومحدد حسب قدرة الخادم.

مثال عملي:

```bash
php artisan queue:work --sleep=3 --tries=3
```

وعند النشر على السيرفر يمكن تشغيل عدد محدود من العمليات مثل 5 Workers.

الفائدة:

- منع استهلاك المعالج والذاكرة بشكل مفرط.
- الحفاظ على استقرار الاستجابة عند الضغط.
- توزيع المهام الثقيلة على عدد من العمال فقط.

### 3) Rate Limiting

يُستخدم لتقليل عدد الطلبات المتكررة على المسارات الحساسة مثل:

- تسجيل الدخول
- إضافة المنتج إلى السلة
- تنفيذ الطلب

المشروع يحتوي أصلًا على تنظيم للـ API throttling عبر:

- `app/Providers/RouteServiceProvider.php`
- `app/Http/Kernel.php`

لذلك يمكن تشديد القيود على المسارات الحساسة بدل إضافة منطق معقد في كل Controller.

### 4) Backpressure بسيط

عندما يرتفع الضغط على النظام، يمكن رفض بعض الطلبات مؤقتًا برسالة واضحة مثل:

```json
{
  "message": "Server busy"
}
```

هذا الحل مناسب فقط كطبقة حماية إضافية، وليس كبديل عن Redis أو تحديد عدد العمال.

## ماذا يؤجل لاحقًا

هذه التحسينات مفيدة لكنها ليست مطلوبة الآن لمسار الطلب الأساسي:

- توزيع المهام حسب الأولوية `high` و `low`.
- Batch processing للمهام الضخمة.
- Load balancing المتقدم.
- نظام مراقبة متكامل للـ queue workers.

السبب:

- المتطلب الحالي يركز على التحكم بالسعة والاستقرار.
- المشروع صغير نسبيًا، ولا يحتاج تعقيدًا إضافيًا الآن.
- الأولوية هي حماية الأداء ومنع الضغط الزائد بأقل تغيير ممكن.

## كيف ينعكس ذلك على المشروع الحالي

في مشروعك الحالي:

- مسار checkout ما زال ينفذ بشكل مباشر داخل `app/Services/CheckoutService.php`.
- الإعدادات الافتراضية ما زالت `sync` للـ queue و `file` للـ cache.
- لذلك Redis + workers محدودة + rate limiting هي أفضل نقطة بداية واقعية.

## الخلاصة

أفضل حل لتطبيق المتطلب الثاني على هذا المشروع هو:

1. تفعيل Redis للـ queue والـ cache.
2. تشغيل عدد محدود من Queue Workers.
3. تفعيل rate limiting على المسارات الحساسة.
4. ترك المهام غير الحرجة لتعمل في الخلفية.

هذا يحقق المطلوب الأكاديمي والعملي بدون توسيع المشروع أكثر من اللازم.
